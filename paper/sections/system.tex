\section{System Overview}
\label{sec:system}

\textsc{TaxonomyAgent} treats taxonomy discovery as a tool-use
problem. An orchestrator LLM proposes typed edits to a working
taxonomy, a parallel judge LLM scores items against that taxonomy and
reports a don't-fit rate, and the loop terminates once that rate falls
below a configurable threshold, gated by a minimum-iteration floor.
This section walks the architecture
(\S\ref{sec:system:arch}), the tools and typed revise operations together with
the convergence signal (\S\ref{sec:system:tools}), and the
operational safeguards that make the pipeline usable as a single
command (\S\ref{sec:system:safeguards}).
The user supplies two inputs: a corpus of texts and a one-sentence
\emph{goal instruction} naming the axis of variation the taxonomy
should discover. For 20~Newsgroups the goal instruction is
``Identify the topic of each text''; for DarkBench it is ``Group
these user prompts by the type of manipulative dark pattern each one
is designed to elicit from an AI assistant.''
The goal instruction is interpolated into the orchestrator's system
prompt verbatim and into every judge classify call; an empty goal
falls back to topic discovery. Figure~\ref{fig:arch} summarises the loop.

\subsection{Architecture}
\label{sec:system:arch}

Two roles are kept strictly separate.

\paragraph{Orchestrator.}
A ReAct-style~\citep{yao2022react} agent built on LangGraph's
\texttt{create\_react\_agent}~\citep{langgraph}; it holds the working
taxonomy as closure state and emits tool calls to inspect the corpus,
edit the taxonomy, or dispatch work to the judge.

\paragraph{Judge.}
A stateless function that labels one item at a time against the
taxonomy the orchestrator hands it; it never sees the taxonomy
history or the orchestrator's reasoning. A weak orchestrator cannot
bluff a label past the judge because the judge derives it from item
text against the current taxonomy. The orchestrator's context length
does not grow with the number of items, only with the number of
taxonomy edits. The judge is trivially parallelisable via a thread
pool.

Both LLMs are called through OpenRouter, so any chat-completions model
supported there can fill either role (\S\ref{sec:evaluation} uses
DeepSeek-v4-Flash and gemini-3.1-flash-lite). Internally, the orchestrator
runs as a LangGraph state machine whose closure holds the working
taxonomy, the set of sampled item ids, a snapshot of the last
finalised taxonomy, and a budget counter for classify calls; because the
taxonomy lives only in this closure, the orchestrator never passes it
as a tool argument, which shrinks payloads and rules out edits to
stale copies.

\subsection{Tools, Typed Revise Operations, and Convergence}
\label{sec:system:tools}

The orchestrator's action space is six tools:
\texttt{sample\_items}, \texttt{get\_taxonomy},
\texttt{revise\_taxonomy}, \texttt{classify\_with\_judge},
\texttt{propose\_novelties\_with\_judge}, and
\texttt{finalize\_classify}. Classification and novelty proposal are
separate verbs because the two judge prompt shapes differ (a single
category with an \texttt{"other"} escape hatch vs.\ open-ended
generation), and conflating them hurt judge reliability in early
prototypes.

\paragraph{Typed revise operations.}
Every taxonomy mutation flows through \texttt{revise\_taxonomy}, which
accepts an ordered list of six typed operations: \textsc{add} a
category, \textsc{rename} one, \textsc{edit} its definition,
\textsc{merge} several into one, \textsc{split} one into several, and
\textsc{drop} a category. Each is dispatched to a pure handler of
signature \texttt{(tax, op)$\to$(new\_tax, log)} under two invariants,
\emph{validate before mutate} and \emph{per-op failure isolation},
detailed in \S\ref{app:dispatcher}. The verb set is standard
taxonomy-edit vocabulary (cf.\ TaxoAdapt~\citep{taxoadapt2025}); the
contribution here is the typed dispatcher with these invariants.

\paragraph{Convergence.}
Stopping the loop requires a cheap signal that the working taxonomy
already covers the corpus. After each \texttt{classify\_with\_judge}
call the system computes the don't-fit rate (the fraction of items the
judge labelled \texttt{"other"}). The orchestrator may call
\texttt{finalize\_classify} once that rate falls below a threshold
(default $0.10$) and the run has cleared a \texttt{min\_iterations}
floor (default $3$). The $0.10$ default is a usability choice we report
alongside external metrics without claiming it is a calibrated leading
indicator (\S\ref{sec:limitations}). Escape-hatch coercion (so a high
don't-fit rate reflects coverage gaps, not hallucinated labels), the
soft-error semantics of the floor, and the isolation of judge errors
from the denominator are detailed in \S\ref{app:convergence}.

\subsection{Operational Safeguards}
\label{sec:system:safeguards}

Three engineering choices make the system usable as a single-command demo.

\paragraph{Cost tracking.}
A \texttt{CostTracker} receives usage from every orchestrator and
judge call and writes \texttt{cost.json} after each agent super-step,
preferring OpenRouter's native \texttt{usage.cost} over a static
price table.

\paragraph{Partial-progress persistence.}
Partial progress is streamed to disk
(\texttt{taxonomy\_state.json} after every revise,
\texttt{classifications.jsonl} row-by-row under a write lock during
finalisation, and \texttt{trace.jsonl} for after-the-fact
inspection), so a process death does not lose the latest taxonomy or
labels.

\paragraph{Streamlit wrapper.}
A single-file Streamlit app (\texttt{app.py}) wraps \texttt{run()}
for non-CLI users, exposes the parameters that matter, streams the
working taxonomy live, and reads the same on-disk artifacts the CLI
writes so runs are inspectable without re-execution
(Figure~\ref{fig:demo}).
