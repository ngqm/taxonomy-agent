\section{System Overview}
\label{sec:system}

\textsc{taxonomy\_agent} treats taxonomy discovery as a tool-use
problem. An orchestrator LLM proposes typed edits to a working
taxonomy, a parallel judge LLM scores items against that taxonomy and
reports a don't-fit rate, and the loop terminates once that rate falls
below a configurable threshold, gated by a minimum-iteration floor.
The user supplies two inputs: a corpus of texts and a one-sentence
\emph{goal instruction} naming the axis of variation the taxonomy
should discover. For 20~Newsgroups the goal instruction is
``Identify the topic of each text''; for CoT-Pattern it is ``Group
these chain-of-thought reasoning traces into categories based on the
dominant style, approach, or behavior visible in the reasoning.''
The goal instruction is interpolated into the orchestrator's system
prompt verbatim and into every judge classify call; an empty goal
falls back to topic discovery.

\subsection{Architecture}
\label{sec:system:arch}

Two roles are kept strictly separate. The \textbf{orchestrator} is a
ReAct-style~\citep{yao2022react} agent built on LangGraph's
\texttt{create\_react\_agent}~\citep{langgraph}; it holds the working
taxonomy as closure state and emits tool calls to inspect the corpus,
edit the taxonomy, or dispatch work to the judge. The \textbf{judge}
is a stateless function that labels one item at a time against the
taxonomy the orchestrator hands it; it never sees the taxonomy history
or the orchestrator's reasoning. The split has three architectural
consequences: a weak orchestrator cannot bluff a label past the judge
because the judge derives it from the item text against the current
taxonomy, the orchestrator's context length does not grow with the
number of items (only with the number of taxonomy edits), and the
judge is trivially parallelisable via a thread pool.

Both LLMs are called through OpenRouter, so any chat-completions model
supported there can fill either role (\S\ref{sec:evaluation} reports
DeepSeek/DeepSeek and Sonnet/Sonnet pairs). Internally, the orchestrator
runs as a LangGraph state machine whose closure holds the working
taxonomy, the set of sampled item ids, a snapshot of the last
finalised taxonomy, and a classify-call budget counter; because the
taxonomy lives only in this closure, the orchestrator never passes it
as a tool argument, which shrinks payloads and rules out edits to
stale copies.

\subsection{Tools, Revise DSL, and Convergence}
\label{sec:system:tools}

The orchestrator's action space is six tools:
\texttt{sample\_items}, \texttt{get\_taxonomy},
\texttt{revise\_taxonomy}, \texttt{classify\_with\_judge},
\texttt{propose\_novelties\_with\_judge}, and
\texttt{finalize\_classify}. Classification and novelty proposal are
separate verbs because the two judge prompt shapes differ (a single
category with an \texttt{"other"} escape hatch vs.\ open-ended
generation), and conflating them hurt judge reliability in early
prototypes.

\paragraph{Revise DSL.}
Every taxonomy mutation flows through \texttt{revise\_taxonomy}, which
accepts an ordered list of typed operations
(\textsc{add}/\textsc{rename}/\textsc{edit}/\textsc{merge}/\textsc{split}/\textsc{drop})
dispatched to pure handlers of signature
\texttt{(tax, op)$\to$(new\_tax, log)}. Two invariants matter for
correctness. \emph{Validate before mutate}: each handler checks every
precondition (source category exists, target name unused, child set
non-empty for \textsc{split}, parents disjoint for \textsc{merge})
before constructing the new taxonomy, so a malformed
\textsc{merge} or \textsc{split} returns the input unchanged with a
log entry the orchestrator can read on the next step (two early bugs
mutated the working taxonomy before all preconditions had been
checked, which motivated this invariant). \emph{Per-op failure
isolation}: the dispatcher applies ops left-to-right, recording
failures per op and continuing with the rest of the batch, so a
bad \textsc{merge} in a three-op call does not roll back a good
\textsc{add} and subsequent ops see the post-prefix state. The verb set itself is standard taxonomy-edit
vocabulary (cf.\ TaxoAdapt~\citep{taxoadapt2025}); the contribution
here is the typed dispatcher with these two invariants.

\paragraph{Convergence.}
After each \texttt{classify\_with\_judge} call the system computes the
don't-fit rate (the fraction of items the judge labelled
\texttt{"other"}). The orchestrator is allowed to call
\texttt{finalize\_classify} once that rate falls below a threshold
(default $0.10$) and the run has cleared a \texttt{min\_iterations}
floor (default $3$). Classify prompts carry a fixed suffix
instructing the judge to reply \texttt{"other"} whenever no listed
category fits, and the \texttt{classify\_with\_judge} wrapper coerces
any out-of-taxonomy reply to \texttt{"other"} before returning to the
orchestrator, so a high don't-fit rate reflects coverage gaps and
not hallucinated labels. The $0.10$ default is a usability choice
that we report alongside external metrics in \S\ref{sec:evaluation}
without claiming it is a calibrated leading indicator
(\S\ref{sec:limitations}). Escape-hatch coercion, the floor's
soft-error semantics, and judge-error isolation from the denominator
are detailed in Appendix~A.

\subsection{Operational Safeguards}
\label{sec:system:safeguards}

Three engineering choices make the system usable as a single-command demo.
A \texttt{CostTracker} receives usage from every orchestrator and
judge call and writes \texttt{cost.json} after each agent super-step,
preferring OpenRouter's native \texttt{usage.cost} over a static price
table. Partial progress is streamed to disk
(\texttt{taxonomy\_state.json} after every revise,
\texttt{classifications.jsonl} row-by-row under a write lock during
finalisation, and \texttt{trace.jsonl} for after-the-fact inspection),
so a process death does not lose the latest taxonomy or labels. A
single-file Streamlit app (\texttt{app.py}) wraps \texttt{run()} for
non-CLI users, exposes the parameters that matter, streams the
working taxonomy live, and reads the same on-disk artifacts the CLI
writes so runs are inspectable without re-execution.
